# Split-K (Flash-Decoding) for B1 H2 S40000 — Design

**Date:** 2026-06-02
**Branch:** s4k-opt · task s4k-perf
**Status:** design sections approved; subagent spec-review applied (B1-B5, N1-N5 fixed 2026-06-02);
pending final user approval before writing-plans
**Goal:** beat CK by 20-30% on the customer shape **B1 H2 S40000 d64 mask0** (and build the
symmetric mask1 path). Baseline native/CK ~98%; this is a structural change, not a leaf tweak.

> Note: this spec is a working artifact on disk. Per user preference it is NOT git-committed.

## 1. Motivation (measurement-backed)

Profiling + asm-analysis + an H-sweep experiment established the limiter is **work-starvation**,
not LDS, registers, or leaf scheduling:

- Grid for B1H2S40000 = `(q_heads=2, m_tiles=313, batch=1)` = **626 blocks = 2.06 blocks/CU**
  (304 CUs). Machine capacity = min(VGPR 512/128=4, LDS 64KB/13.5KB=4) = **4 blocks/CU**. The
  CU is half-filled by *work supply*, not resource limits. The big axis — the 625-iteration
  serial KV reduction inside each block — is the one axis NOT parallelized.
- **H-sweep (fixed per-block work, vary #blocks):** TFLOPS 258.7 (H1) → 337.5 (H2, customer) →
  427.9 (H16) → 423.5 (H32 plateau). **+65%** purely from more independent blocks ⇒
  work-starvation confirmed; customer-shape headroom ≈ **+27%** (337→428).
- **Overhead-tax sweep (CU saturated, shrink KV loop to emulate split factor G):** no penalty
  down to ~157 tiles/block, −3% at 79 tiles (G=8), −15% at 40 tiles (G=16).
- **Combined estimate:** G≈8 → 16.5 blocks/CU (saturated) with ~3% per-split tax ⇒ **~+20-22%
  before the combine pass**. Lower edge of the goal; reaching 30% needs split-K + a within-block
  ILP win later.

Verified non-levers (do not pursue): LDS layout (bank conflicts already ≈0), VGPR reduction
(raises ceiling, not resident count, until more blocks exist), leaf GEMM reorders.

## 2. Architecture

**Two-pass split-K, additive, mask0+mask1, fixed G=8, sharing the proven hot loop via an
`if constexpr (IsSplit)` template parameter.**

```
Pass 1 (split):   grid(q_heads, m_tiles, batch*G)    each block walks ~ceil(625/G) KV tiles
                  → writes partial output O_g (fp32) + partial LSE_g (fp32) to scratch
Pass 2 (combine): grid(q_heads, m_tiles, batch)       reads G partials per (b,h,m_tile),
                  → reweights by LSE, bf16-truncates once, writes final O (+ optional LSE)
```

- **Split contract (minimal):** each split emits ONLY a **normalized partial output** `O_g`
  and a **partial LSE scalar** `LSE_g` per row. No separate max/sum. This is the standard
  FlashAttention split-KV interface and reuses the LSE value `epilog_store` already computes.
  **Base, pinned (do NOT write `scale` unqualified):** `epilog_store` (op_epilog.hpp:92-96)
  writes the NATURAL-log value `LSE_g = (log2(l_g) + params.scale·m_g)·ln2 = ln(l_g) + sm_scale·m_g`,
  where `params.scale` is the **base-2 (log2e-folded)** scale (params.hpp:38-42) and
  `sm_scale = params.scale·ln2` is the plain `1/sqrt(d)` softmax scale. `__builtin_amdgcn_logf`
  emits `v_log_f32 = log2` (op_gemm.hpp:43-45), hence the trailing `·ln2`. The split path stores
  this natural-log `LSE_g` to scratch unchanged.
- **Combine math** — runs in **natural-e** on the stored natural-log `LSE_g` (decision: option (a)):
  `L* = ln(Σ_g exp(LSE_g − M)) + M` with `M = max_g LSE_g`; `O = Σ_g exp(LSE_g − L*)·O_g`.
  Weights sum to 1 by construction — pure reweight, no numerator rescale. Implement with natural
  `expf`/`logf` (NOT exp2/v_log_f32) since `LSE_g` is already in natural-log domain; or, if the
  device combine prefers base-2 builtins, first convert (`LSE2_g = LSE_g·log2e`) and use
  `exp2`/`v_log_f32` consistently on BOTH the weight and `L*`. The base MUST be consistent across
  `LSE_g`, the `exp`, and the `log` — a base-e-vs-base-2 slip reconstructs the wrong O and slips
  under the loose end-to-end tolerance (§5 names this risk; the isolated combine test §5.1 catches it).
- **Partial precision:** `O_g` written **fp32** to scratch → combine reweights in fp32 →
  **single** bf16 truncation at the end (matches the single-pass kernel's one-truncation path).
- **Shared device fn:** `fmha_fwd_d64_device<HasMask, IsVarlen, IsSplit>` (`__forceinline__` into
  each `__global__`). With `IsSplit=false` every split branch folds via `if constexpr`. **Safety
  gate (B4):** the existing 4 entries must be **functionally identical** — same hot-loop instruction
  counts/mix (G-MIX `rocprofv3 --pmc`) AND the bit-exact golden gate (G-EXISTING) stays green. A
  `.s` diff is kept as an *informational* check, NOT pass/fail: adding a 3rd template param + a
  discarded `if constexpr` branch under `-O3` (with the load-bearing `-mllvm` flags) can perturb
  block ordering/RA by a byte without changing behavior, so byte-identical is too brittle a bar.

## 4. Data flow & scratch (B1 H2 S40000 mask0, G=8)

```
Pass 1: grid(2, 313, 8)  = 5008 blocks (16.5/CU)
  block(b,h,m,g) walks KV tiles [g·79 .. min((g+1)·79, 625))  (T = ceil(625/8) = 79)
  → scratch_O  [g][b][h][row][64]  fp32   (split-major)
  → scratch_LSE[g][b][h][row]      fp32
Pass 2: grid(2, 313, 1)  combine
  block(b,h,m) reads 8×(O_g, LSE_g) for its ≤128 rows, reweights, truncates → final O
```

**Layout rationale (split-major `[g][b][h][row][64]`), inner row = natural head-dim order:**
The inner `[b][h][row][64]` must store each row in **natural head-dim order** — which requires
`epilog_store_split` to apply the **same `swz` column un-swizzle** the current `epilog_store` uses
(op_epilog.hpp:128-131; `swz` swaps bits 2,3, op_gemm.hpp:59-68). The O_acc registers carry the
SwizzleA head-dim ordering inherited from GEMM1; `swz` is what lands them natural-order in DRAM. If
the rewritten fp32 store (B2) drops `swz`, scratch rows are permuted and the combine reads garbage.
So: **split store keeps `swz`; combine reads plain natural-order `[g][b][h][row][d]` fp32.** Test
§5.1 Source-B (real partials vs full attention) is the assertion that catches a dropped/extra swz.
NOTE on the per-lane mapping: `64` is per-ROW data, NOT per-lane — under `TransposedC` each lane
owns one `(row, k_sub-half)` and writes 32 fp32 (`o_acc_d0[16]+o_acc_d1[16]`); the 64-dim row is
filled by a lane pair (`k_sub` 0 and 1). Address/ownership math is reused from `epilog_store`; only
the store *width/stride/pack* changes (B2). The outer `[g]` ordering choice: **split-major** keeps
each split's output one contiguous O-image and lets the combine loop `g=0..7` reading 8 planes; the
split-minor alternative `[b][h][row][g][64]` makes one row's G partials contiguous (better for a
serial combine) but scatters each split kernel's writes. Default split-major; the combine read is
plane-strided (8 planes × per-row) — re-check against measured combine BW during impl.

**`epilog_store_split` fp32 thread mapping is a TUNABLE, not a given:** `epilog_store`'s lane→address
mapping is tuned for a **64 B/lane** bf16 write (8×b64). fp32 doubles this to **128 B/lane** (16×b64
or 8×dwordx4), so the coalescing that is optimal for the bf16 store is NOT assumed optimal for fp32.
The mapping (how the 32 fp32/lane map to scratch bytes — contiguous-per-lane vs interleaved, store
width b64 vs dwordx4) must be **measured and tuned** for the fp32 scratch write, which is the Pass-1
tax. Gated: G-SOL checks the split kernel's store/VMEM behavior and G-ASM checks the store
instruction selection on the new fp32 path (see §7). Start from the reused mapping; optimize if the
split-store VMEM/BW shows up in G-SOL.

**Scratch sizing:** `O_partials = G·B·q_heads·seq_len·64·4 B`; `LSE_partials = G·B·q_heads·seq_len·4 B`.
At G=8/B1/H2/S40000: `8·1·2·40000·64·4` = **163.84 MB** O + 2.56 MB LSE. Allocated once by the
harness (bench/test); scales with G·B·H·S. Trivial on MI300X (192 GB).

**Scratch traffic (a real cost the sweeps did NOT include):** ~164 MB write (Pass 1) + ~164 MB read
(Pass 2) ≈ **328 MB**. At an **estimated** ~3.5 TB/s effective HBM (≈66% of MI300X ~5.3 TB/s peak;
reasonable-to-slightly-optimistic for streaming, and the Pass-2 read is plane-strided so less ideal
than the contiguous write) ≈ **~0.094 ms ≈ 3.6%** of the 2.6 ms baseline; even at a pessimistic
2.5 TB/s it is ~0.13 ms ≈ 5%, still in-band. Neither the H-sweep nor the overhead-emulation sweep
paid this, so it is an additional tax on top of the ~+20-22% estimate — acceptable, but it is why
realized gain lands at the lower edge, not the full +27%. Confirm the real cost in G-BENCH.

## 5. Verification — oracle layers + a two-stage trust chain

**Trust chain (why we add GPU refs):** the CPU refs (`cpu_ref_split`, `cpu_ref_combine`) are the
*ground truth* but are minutes-slow at S40000. So we establish trust at SMALL S, then verify the
kernel at LARGE S with a fast GPU ref:

```
Stage 1 (small S, establish trust):  CPU ref  ==  GPU ref     (gpu_ref_split / gpu_ref_combine
                                                                 validated against cpu_ref_split /
                                                                 cpu_ref_combine — prove the GPU ref)
Stage 2 (large S incl. S40000, fast): GPU ref  ==  our kernel  (the now-trusted GPU ref verifies
                                                                 the split & combine kernels quickly)
```

This unlocks large-S **isolated** verification (split-alone, combine-alone) that pure-CPU could only
do at small S — and lets us add many more / larger test cases (§5.2).

| Layer | Device under test | Oracle | Inputs |
|---|---|---|---|
| **GPU ref trust** | `gpu_ref_split`, `gpu_ref_combine` | `cpu_ref_split`, `cpu_ref_combine` (CPU) | small S, real + synthetic |
| **Split alone** | split kernel `O_g`,`LSE_g` for one range | `cpu_ref_split` (small S) / `gpu_ref_split` (large S) | real Q/K/V |
| **Combine alone** | `combine_split` / `fmha_fwd_d64_bf16_combine` | `cpu_ref_combine` (small S) / `gpu_ref_combine` (large S) | synthetic + real partials |
| **End-to-end** | split+combine | `cpu_ref_verify` (small S) / `gpu_ref_fmha_fwd` (large S, incl. S40000) | real Q/K/V |

- **Why a standalone combine oracle:** the combine is a tiny fraction of end-to-end FLOPs, so a
  reweight/precision bug hides under the overall tolerance. The isolated test feeds *synthetic*
  partials so the bug cannot be masked by plausible split output. Stress cases: widely different
  `LSE_g`; one split dominating (`LSE_g` ≫ rest → weight→1); fully-masked split (`LSE_g=−inf` →
  weight 0, no NaN); **G=1 identity** → output equals the plain epilogue exactly.
- **Why extend cpu_ref (not a new split oracle):** one oracle serves full + partial attention.
  The refactor is behavior-preserving: `cpu_ref_verify` calls `cpu_ref_split` with
  `kv_start=0, kv_end=Skv, lse_out=nullptr`. **De-risk:** run the existing 67 fused + 49
  standalone tests before/after the refactor — must stay green.
- **G=1 split check — split into TWO gates (B5):**
  - (a) **G=1 split-store, fp32:** the split's **fp32** `O_g` must equal the single-pass epilogue's
    fp32 accumulator **before bf16 truncation**, bit-exact. This is achievable and isolates the
    KV-range-narrowing logic from combine. (Compare against an instrumented single-pass that dumps
    the pre-trunc fp32 O, or against `cpu_ref_split` full-range fp32.)
  - (b) **G=1 end-to-end through combine:** **within tolerance**, NOT bit-for-bit. Even at G=1 the
    combine computes `L*` from `LSE_1` via a `logf`/`expf` round-trip, so the weight is `1.0f ± 1 ulp`
    and the final truncation need not match. To make (b) as tight as possible, the combine's bf16
    truncation MUST reuse the identical `epilog_store` selector `kBf16TruncSel = 0x07060302`
    (op_epilog.hpp:103); optionally special-case `num_splits==1` to skip the reweight entirely.

### 5.1 Test-case enrichment — two complementary partial sources

A GPU-vs-oracle test only proves *the kernel matches the oracle*; it does NOT prove *the oracle
itself is correct*. So `test_combine` draws partials from TWO sources:

- **Source A — synthetic generator (kernel-vs-oracle):** a seeded, parameterized matrix the test
  controls, chosen to hit adversarial numerics real data rarely produces. Asserts
  **GPU `combine_split` == `cpu_ref_combine`**. Axes:
  - `G` ∈ {1, 2, 3, 4, 5, 7, 8, 16} — **incl. non-powers-of-2 and counts that don't divide the
    tile count**, to exercise ragged partitions and degenerate-empty splits (G-INVARIANCE).
  - `LSE_g` distribution: all-equal · one-dominant · monotone ramp · random ± · some splits `−inf`
  - row population: full 128 · partial 64 (last M-tile) · single row
  - multi-head
- **Source B — real-partial cross-check (oracle-and-partition validation, the G-INVARIANCE test):**
  for EACH `G` in the set above, generate partials by running `cpu_ref_split` over `G` contiguous KV
  ranges on REAL Q/K/V, then assert **`combine(partials)` == `cpu_ref_verify`(full range)** within
  tolerance — i.e. the output is invariant to G. This ties split+combine back to plain full
  attention, catching KV-range-partition / ragged-split / LSE-domain bugs that Source A cannot
  (synthetic `LSE_g` has no ground-truth `O` beyond the reweight identity).

A pulls the kernel toward the oracle; B pulls the oracle (and the partition) toward ground truth.
Both must pass.

### 5.2 Enriched test-case set (enabled by the fast GPU ref)

Because Stage-2 verification (GPU ref == kernel) is fast even at S40000, we can test a much wider
matrix than the CPU-only path allowed. The set, split by what each case targets:

**Shape / partition axes** (split & combine, large S via GPU ref):
- **S (KV length):** 2048, 4096, 8192, 16384, **40000** (customer), 40064 (exact 313·128, no Q
  tail), 39936 (312·128, different tail). Catches S-dependent scratch-offset / overflow bugs the
  small-S CPU path can't reach.
- **S not a multiple of kN0=64:** e.g. 40000 (625·64 exactly — clean), 40001, 39999 (last KV tile
  partial) — exercises the partial-last-KV-tile boundary interacting with the split range.
- **G:** {1,2,3,4,5,7,8,16,32} incl. non-divisors of the tile count and **G > tile count** (forces
  many empty splits → all-but-few `LSE_g=−inf`).
- **Q tail:** S where the last M-tile is half-empty (40000 = 312·128+64) AND S a clean multiple of
  kM0=128 (40064) — the row-bounds guard must hold both ways.

**Numeric / masking axes** (combine, mostly synthetic Source A + real Source B):
- mask0 and mask1 (causal: a split entirely in the masked-future region must yield `LSE_g=−inf`).
- all-`−inf` row (every split masked) → zeros, no NaN.
- one split dominating; LSE values spanning a wide exponent range (over/underflow of `exp(LSE−L*)`).

**Heads:** B1H1, B1H2 (customer), multi-head B1H8, and B2+ to confirm the `batch*G` z-mapping
(`b = blockIdx.z / G`, `split_idx = blockIdx.z % G`) decodes correctly for B>1.

**Trust-chain cases** (Stage 1, small S so CPU is fast): `gpu_ref_split` vs `cpu_ref_split` and
`gpu_ref_combine` vs `cpu_ref_combine` across a few (S,G,mask) points — these MUST pass before any
Stage-2 result is trusted; a Stage-2 pass with an unvalidated GPU ref proves nothing.

**Edge handling:**
- **Empty trailing split** (range past seqlen, e.g. large `g`): the device fn's existing
  `num_total_loop <= 0` degenerate path (pipeline.hpp:207-222) fires — but it currently calls
  `epilog_store` (bf16). The split path MUST route the degenerate case through
  `epilog_store_split` so the **fp32** scratch plane gets the `LSE_g=−inf` + zero-O sentinel; else
  the combine reads an uninitialized/bf16 plane.
- **Fully-masked row** (mask1) → all `LSE_g=−inf` → combine emits zeros (matches the existing
  `rsum==0` path, op_epilog.hpp:77).
- **Combine row guard:** the combine kernel is a NEW entry and MUST apply the same
  `abs_m_row < seqlen_q` row-bounds guard as `epilog_store` (op_epilog.hpp:136) when writing final
  O, and must not read scratch rows past `seqlen_q`.
- **G∤625** → last split short via the `T=ceil(625/G)` ceiling.

## 6. Scope & sequencing (TEST-FIRST)

Build **oracle → failing test (red) → kernel (green)** per layer. No kernel is written before its
test exists and fails. Order:

1. **Oracles (pure CPU, host-testable, no GPU):**
   - extend `cpu_ref` → `cpu_ref_split(kv_start, kv_end, lse_out)`; `cpu_ref_verify` becomes a
     full-range caller (behavior-preserving — existing tests must stay green: gate G-EXISTING).
   - add `cpu_ref_combine` (CPU combine oracle).
2. **Combine test (RED):** write `test_combine` against `cpu_ref_combine` with the §5.1 two-source
   matrix (synthetic A + real-partial B). Fails — no kernel yet.
3. **Combine kernel (GREEN):** implement `combine_split` + `fmha_fwd_d64_bf16_combine` → G-COMBINE.
4. **Split test (RED):** G=1-identity (bit-exact vs single-pass) + one-range vs `cpu_ref_split`.
5. **Split kernel (GREEN):** add `IsSplit` param to `fmha_fwd_d64_device` + `epilog_store_split` +
   `_msk0_split`/`_msk1_split` entries → G-SPLIT; `G-NOREG-ISA` guards the non-split `.s`.
6. **Bench wiring + E2E last:** `--splitk G` two-pass + scratch alloc; measure speedup at S40000
   (G-E2E, G-BENCH) vs the H-sweep estimate (~+20-22%). **No S40000 gtest config exists**
   (`test_configs.hpp` tops out at S8192), so G-E2E runs through the **bench `--verify`** path
   (split+combine vs `gpu_ref_fmha_fwd` on-device), NOT the gtest suite. `cpu_ref_verify` at
   B1H2S40000 is a ~minutes-long host computation — use it only as a spot check; the §5.1 Source-B
   real-partial correctness anchor should run at a **reduced S** (e.g. S≤4096) where the host oracle
   is fast, since the combine math is S-independent.

Golden-gate integration (tolerance-based for the split path) deferred until the bench win is
confirmed. mask1 path built symmetric; its **performance tuning** is lower priority (mask1 S40000
already wins ~127%), but its **correctness is NOT optional** — G-E2E and §8 both require
`--mask 1 --splitk 8 --verify` to pass. This is load-bearing on one invariant: under causal masking,
a split whose entire KV sub-range lies in the masked-future region produces a fully-masked tile, and
it MUST still write its scratch plane with the `LSE_g=-inf` + zero-O sentinel (the degenerate
`num_total_loop<=0` path routed through `epilog_store_split`, see §5 "Empty trailing split"). That
sentinel is what makes the combine partition-agnostic for causal. "Lower priority" therefore means
*we don't chase the mask1 split speedup first*, NOT *we skip mask1 correctness*.

**Out of scope:** varlen split-K; within-block ILP wins (PERF-002/005); G auto-tuning heuristic
(fixed G=8 for now, though `--splitk G` allows live sweeping).

## 7. Quality gates

**Reuses the established per-task gate protocol** from
`docs/superpowers/specs/2026-06-01-fmha-gap-fixes-design.md` §3 (`G-CORRECT / G-REG / G-MIX /
G-SOL / G-BENCH`, "fail any gate → revert, not patch forward"), PLUS the asm-level
**ISA-verification gate** from `docs/superpowers/specs/2026-05-31-phase3-isa-parity-benchmark-design.md`
§"Gate 4" (`/amdgpu-asm-analyzer` CFG + dependency-table). That protocol assumed a single bit-exact
kernel; split-K adds a new non-bit-exact code path + two new kernels + scratch, so the gates are
adapted below. Both profiling gates are carried forward and are load-bearing here: **G-SOL** is the
same rocprof-compute tool that diagnosed the work-starvation lever, so it is how we prove the fix
*landed* (not just that the bench moved), and **G-ASM** is the same asm-analyzer (CFG + dep table)
we used to rule out leaf bottlenecks — now applied to codegen we have never seen (two new kernels).
The intent — every task independently verified, no task done until it passes — is unchanged.

**Every gate states GOAL (what it proves) · METHODOLOGY (how) · COMMAND (exact, runnable) ·
PASS (the criterion).** All commands assume the native container `poyenc-fmha` with the repo
bind-mounted at `/root/workspace`, GPU2 (`HIP_VISIBLE_DEVICES=2`), GPU confirmed idle first via
`python3 ~/.claude/skills/gpu-usage/scripts/gpu_usage.py --format markdown`.

#### G-EXISTING  (mandatory, every task)
- **Goal:** no edit regresses the proven kernels; existing correctness is bit-exact-preserved.
- **Methodology:** the full build+golden gate (67 fused + 49 standalone, bit-exact vs CK, `.s` extract).
- **Command:** `docker exec poyenc-fmha bash -c "/root/workspace/scripts/run-gates.sh"`
- **Pass:** `[  PASSED  ] 67 tests.` + seven `[  PASSED  ] 7 tests.` + `ALL REQUESTED GATES PASSED`, 0 SKIPPED.

#### G-REFSPLIT  (T1 — host oracle, range + LSE)

- **Methodology:** a host-only check (throwaway `main` or `test_cpu_ref_split`, no GPU) on small inputs.
- **Command:** `docker exec poyenc-fmha bash -c "cd /root/workspace && <build+run the host cpu_ref_split check>"`
- **Pass (4 asserts):** (1) full-range O == legacy inlined O bit-exact; (2) two complementary ranges → `cpu_ref_combine` == full-range O within 1e-4; (3) LSE on a hand-computed 2-key input == `logf(sum)+scaled_max` to 1e-5 (FAILS if `*scalar` re-added — the A1 regression test); (4) empty range → `LSE=-inf`, `O=0`.

#### G-COMBINE-HOST  (T2 — host oracle sanity)
